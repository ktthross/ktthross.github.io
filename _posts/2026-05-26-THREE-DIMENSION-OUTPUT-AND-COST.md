The first problem I need to solve is how to score my generative model for three dimensional objects. The model will output the vertices and the faces, but once I have those, how do I know if it has learned to generate the object I'm interested in?

My goal is to accomplish this without doing extensive measurements of the object itself to provide labeled data for training. This would be laborious to the point that you should probably just model the object by hand. Now there needs to be some human configuration to limit the size of the model. I suppose you could constrain the model and request that it optimize some objective while minimizing the number of vertices, or some such thing, but that is a step beyond what I am attempting.

We already have ground truth data for our work. We have images of the object we are interested in. If we can create an object that renders in such a way that it reproduces the object in the image from different angles, then we have succeeded. For each pass of the model, render the object, then have a "camera" look at the object from different angles and compare it to the input images. If the rendered object looks like the original photo, we have succeeded.

Hurray! I have just created a bunch more problems! This requires that we know the position of the camera relative to the other images, that we know the distance of the camera to the object, and likely some other information I am not yet thinking of.

One option is to have a set group of distances and orientations. If you don't require all of those angles, you could only include examples that you have and mask out those you don't. This way, if you only need certain angles, you can include those. This makes the model much simpler, but it is also constrained. It puts a large burden on the training data and setup. It also means some features of the object may not be visible if they cannot be resolved from the selected angles.

Another option I can think of is to solve for, or discover, the angles and distances. Imagine, if you will, an object which is already perfectly rendered. How do you match the camera angle to reproduce the image? If we have a solution for this, we could make it part of our cost function.

If we can design our cost function around not only generating the object but also finding the relevant parameters for the camera, we can let the model do the hard work. From the rendering example we went through, for each image the most important thing to learn is the camera position. We potentially could learn the focal length as well as the width and height of the image. However, width and height could be standardized as part of feature generation, and that is likely the first way to try this out.

The cost function would essentially be the difference between the image provided as input and the rendered image. The same vertices and faces would be used for the images put into the model. As you add more photos, it would hopefully learn more of the full 3D space of the object.

```python
vertices, faces, camera_position = model.forward(data)
rendered_image = render_from_obj(vertices, faces, camera_position)

loss += some_sort_of_pixelwise_loss_function(rendered_image, ground_truth)
```

I like this idea. It's a good first pass at a cost function. Something to dig into in the next entry. There are likely a bunch of ways this can go wrong, but that's the fun part. At the very least, this gives me a concrete loop: generate geometry, render it from a guessed camera position, compare it to the target image, and improve the guess.
